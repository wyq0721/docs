## 01-编译性能看板

### 1. 场景概述
- 场景名称：编译性能看板（Compiler Performance Panel）
- 适用对象：编译器性能优化工程师
- 发生频率：每轮性能优化周期均需使用
- 业务重要性：高——性能分析是编译器优化的基础，直接影响优化方向和效果

### 2. 背景与痛点
- 原来是怎么做的：手动分析编译日志和 profiling 数据，缺乏可视化工具，依赖文本搜索和人工比对
- 原来的主要痛点：团队无前端技术栈，无法自建可视化分析工具；依赖周边工程团队协助周期长、沟通成本高
- 为什么值得优化：性能分析是高频刚需，可视化工具能大幅提升分析效率和准确性

### 3. AI 介入点
- AI 用在了哪个环节：整个前端开发过程——从架构设计到代码实现全部由 AI 完成
- 具体承担什么角色：全栈开发者——团队成员担任产品经理，AI 担任前端开发
- 人工仍负责什么：需求定义、功能验收、性能数据的业务解读

### 4. 操作流程
- 第一步：团队成员定义需求——需要什么样的可视化视图、支持什么数据格式
- 第二步：AI 生成前端代码——HTML/CSS/JS + ECharts + Canvas 自定义绘制
- 第三步：迭代优化——根据实际使用反馈调整交互细节和功能
- 第四步：部署上线——Cloudflare Pages（外网）+ Nginx（内网）

### 5. 使用工具与输入输出
- 使用工具：Cursor（AI 辅助编码）
- 输入材料：
  - `*.time.prof` / `*.mem.prof`：编译阶段时间和内存 profiling 数据（JSON）
  - `time_cost.json`：Chrome Trace Event 格式的时间线数据
  - `memory_cost.json`：时序内存数据
  - `package_command.json`：包名到编译命令的映射
- 输出结果：完整的 Web 性能分析看板
- 输出格式：单页 Web 应用，支持离线 HTML 导出

### 6. 实际效果
- 效率变化：从"没有工具只能人肉分析"到"可视化一目了然"，性能分析效率提升数倍
- 质量变化：Perfetto 风格的时间线视图 + 内存曲线同步联动，可精确定位性能瓶颈
- 使用者反馈：团队日常性能分析的标准工具
- 是否形成复用价值：是——已开源，其他编译器团队可直接使用

### 7. 风险与限制
- 主要风险：AI 生成的前端代码为单体架构（~5870 行单文件），后续维护成本较高
- 不适用情况：需要复杂后端交互的场景
- 容易出错的地方：Canvas 自定义绘制的边界条件处理

### 8. 经验总结
- 成功关键：
  - 人当产品经理、AI 当开发——角色分工清晰
  - 零依赖架构（纯 HTML/CSS/JS）降低了 AI 出错概率
  - DJB2 哈希保证包名颜色一致性等细节体现了 AI 的工程能力
- 复用建议：适合所有"团队缺某技术栈但有明确需求"的场景
- 后续优化方向：模块化重构、增加更多分析维度

### 9. 一页摘要
- 这个案例解决了什么问题：团队无前端能力但急需性能可视化工具的矛盾
- AI 的核心价值是什么：消灭路径依赖——让团队不再受限于自身技术栈
- 最值得推广的点是什么："人当 PM、AI 当开发"的协作模式
- 一句话结论：AI 让"不会做"变成"不需要会"，一周内从零搭建完整性能分析平台

## 02-ACE 平台（Agent Centric Engineering）

### 1. 场景概述
- 场景名称：ACE 平台——以 Agent 为中心的研发工程平台
- 适用对象：仓颉编译器前端团队全员，及下游使用团队
- 发生频率：日常研发持续使用
- 业务重要性：极高——团队 AI 辅助研发的核心基础设施

### 2. 背景与痛点
- 原来是怎么做的：个人各自使用 Cursor/Copilot 等工具，经验无法共享，知识库分散
- 原来的主要痛点：单兵作战只能止步于个人效率提升，无法对团队和组织带来进一步变革；个人积累的 prompt 经验和领域知识无法复用
- 为什么值得优化：要实现从个人效率到组织效率的跨越，必须有平台化的解决方案

### 3. AI 介入点
- AI 用在了哪个环节：研发全流程——从 issue 分析、方案设计、代码实现到测试验证
- 具体承担什么角色：多 Agent 协作——Defender（构建/修复）、Attacker（审查/挑战）、Judge（裁决）
- 人工仍负责什么：方案审批、最终 code review、合入决策

### 4. 操作流程
- 第一步：通过对话或 YAML 配置定义工作流（支持 phase-based 线性模式和 state-machine 状态机模式）
- 第二步：Agent 按工作流自动执行，遇到信息缺口通过 `[NEED_INFO]` 协议向其他 Agent 或人类请求
- 第三步：攻防迭代——Defender 产出 → Attacker 审查 → Judge 裁决，不通过则自动回滚重试
- 第四步：关键节点人工审批，通过后 Agent 继续执行直至完成

### 5. 使用工具与输入输出
- 使用工具：ACE 平台（Next.js 16 全栈应用），执行引擎支持 Claude Code CLI、Kiro CLI、OpenCode
- 输入材料：issue 描述、代码仓库、知识库（18 个模块、714 个文件、1036 个类）、7 个仓颉专用 skill
- 输出结果：修复 PR、分析报告、架构文档、测试用例
- 输出格式：代码 + 结构化文档，全链路持久化到 `runs/{runId}/`

### 6. 实际效果
- 效率变化：Issue #701 从分析到修复 16.5 小时自动完成（57 步，含 3 次自动回滚）
- 质量变化：攻防迭代机制显著提升产出质量，#701 最终方案与人类专家完全一致（单行修复 `Attribute::IN_EXTEND`）
- 使用者反馈：团队从"各自为战"转向"平台协作"，知识沉淀效率大幅提升
- 是否形成复用价值：是——工作流模板和 skill 可跨场景复用，平台架构支持多引擎扩展

### 7. 风险与限制
- 主要风险：Agent 在复杂场景下可能陷入无效循环，需要状态机的 maxTransitions 和 circuit breaker 机制兜底
- 不适用情况：需要大量人际沟通和政治判断的工作
- 容易出错的地方：Agent 信息不足时倾向于猜测而非请求信息——Supervisor-Lite 的 Plan Loop 机制正是为此设计

### 8. 经验总结
- 成功关键：
  - 状态机 + 攻防迭代的工作流设计，让 Agent 可以自动回滚和重试
  - Supervisor-Lite 智能路由打破了线性信息流的约束
  - 知识库和 skill 的持续积累形成飞轮效应
- 复用建议：适合所有需要多步骤、多角色协作的复杂研发任务
- 后续优化方向：增强自迭代能力（每次运行自动更新知识库和 skill）、接入更多执行引擎

### 9. 一页摘要
- 这个案例解决了什么问题：从单兵 AI 辅助到团队级 Agent 协作平台的跨越
- AI 的核心价值是什么：平台化沉淀团队知识和经验，让飞轮转起来
- 最值得推广的点是什么：状态机工作流 + 攻防迭代 + Supervisor 智能路由的架构设计
- 一句话结论：ACE 平台让 AI 辅助从"个人工具"升级为"团队基础设施"

### 10. 架构深度解析

#### 10.1 状态机工作流引擎

ACE 的核心引擎是有限状态机（FSM）驱动的工作流执行器，与传统线性流水线有本质区别：

**核心机制：**
- 每个工作流由多个 state 组成，每个 state 包含若干 steps（Agent 执行单元）和 transitions（状态转移规则）
- Agent（通常是 Judge 角色）输出结构化 JSON 判决，如 `{"verdict": "fail", "next_state": "根因分析"}`
- 系统通过三层状态转移决策链确定下一步：① Agent 显式指定 next_state → ② YAML 中配置的 transition condition 匹配 → ③ issueRoutingRules 按 pattern 匹配

**安全保障：**
- `maxTransitions`（默认 50 次）防止死循环
- 自转移（self-transition）有独立的断路器机制
- 崩溃恢复：运行状态持久化到 `runs/{runId}/state.yaml`，服务重启后自动断点续跑

**人工检查点（HITL）：**
- 状态可配置 `requireHumanApproval: true`，执行完成后暂停等待人工审批
- 支持三种操作：通过、继续迭代（附反馈）、拒绝
- 支持 `forceTransition`（强制跳转到任意状态）和 `inject-feedback`（实时反馈注入）

**两种工作流模式对比：**

| 维度 | Phase-based（线性） | State-machine（状态机） |
|------|-------------------|----------------------|
| 流程方向 | 单向前进 | 可回退、可跳转 |
| 迭代控制 | 阶段内 maxIterations | 全局 maxTransitions |
| 适用场景 | 简单线性任务 | 复杂缺陷修复、多轮对抗 |
| 配置复杂度 | 低 | 中（需定义 transitions） |

#### 10.2 Supervisor-Lite 智能路由

**设计初衷：** 在实践中发现，把任务一句话扔给 AI 时，它很容易理解偏差、凭空猜想用户意图。于是产生了需求澄清的想法——让 Agent 跟用户或其他 Agent 来澄清需求，把信息理解透彻并精准传递。这就是 Supervisor-Lite 的起源。

**核心洞察：** Agent 自己最清楚自己缺什么信息，而谁能回答这个问题应该由全局视角的路由器决定。

**职责分离三角：**

| 角色 | 职责 | 明确不做的事 |
|------|------|-------------|
| Agent | 领域专业工作 + 判断自身信息缺口 + 区分"问人"还是"问机器" | 不需要知道团队有哪些 Agent，不判断该问谁 |
| Supervisor（路由器） | 根据问题内容决定转发给哪个 Agent | 不参与业务讨论、不产出任何内容 |
| WorkflowManager | 状态流转、持久化、人机交互暂停/恢复 | 不做路由决策 |

**关键设计决策：** Agent 的 prompt 中不包含团队编制信息。这避免了"Agent 列表注入 → prompt 膨胀 → Agent 尝试自行路由 → 路由质量不可控"的常见陷阱。

**NEED_INFO 协议（极简的 Agent 通信契约）：**

| 标记 | 含义 | 处理路径 |
|------|------|---------|
| `[NEED_INFO]` | 需要技术/专业信息 | Supervisor 路由到最合适的 Agent |
| `[NEED_INFO:human]` | 需要人工确认或业务决策 | 直接暂停执行，前端弹窗请用户回答 |
| `[PLAN_DONE]` | 信息已充分，开始正式执行 | 结束信息收集循环，进入执行阶段 |

协议设计的关键取舍：
- **极简**：只有两种需求类型（技术 vs 人工），Agent 几乎不可能用错格式
- **声明式**：Agent 只声明"我需要什么"，不指定"谁来回答"，路由决策完全集中在 Supervisor
- **Agent 无感知**：Agent 不知道系统中有哪些其他 Agent，保持 prompt 的纯净性和专注度

**两层路由决策：**

```
                Agent 声明信息需求
                       │
                ┌──────▼──────┐
                │ :human 标记？ │
                └──┬──────┬───┘
              是   │      │ 否
  ┌────────────┘      └──────────────┐
  │ 暂停，前端弹窗                    │
  │ 等待用户回答                ┌─────▼─────┐
  └──────────────┐              │ 关键词匹配  │（零 LLM 开销）
                 │              └──┬────┬───┘
                 │            命中 │    │ 未命中
                 │     ┌──────────┘    └──────────┐
                 │     │                    ┌─────▼─────┐
                 │     │                    │ LLM 语义路由 │（~$0.001/次）
                 │     │                    └──┬────┬───┘
                 │     │              命中     │    │ 未命中
                 │     │         ┌────────────┘    └──────┐
                 │     │         │                        │
                 │  ┌──▼─────────▼──┐              ┌─────▼─────┐
                 │  │ queryAgent     │              │ Fallback   │
                 │  │ 临时问答       │              │ 降级给用户  │
                 │  └───────────────┘              └───────────┘
```

**Plan 循环：** 通过 `enablePlanLoop: true` 按步骤粒度启用，`maxPlanRounds`（默认 3）防止无限请求。将 Agent 执行显式拆分为"信息收集"和"正式执行"两个阶段，Agent 在信息充分后才开始产出，显著减少因信息不足导致的返工。

#### 10.3 攻防迭代机制（Defender-Attacker-Judge）

每个工作流阶段可配置三种角色：

- **Defender（蓝队）**：负责构建、修复、实现——产出代码或方案
- **Attacker（红队）**：负责审查、挑战、破坏——寻找 Defender 产出中的漏洞
- **Judge（裁判）**：输出结构化判决（`pass` / `fail` / `conditional_pass`），驱动状态转移

Judge 的判决直接决定工作流走向：
- `pass` → 进入下一状态
- `fail` → 自动回滚到前序状态重试
- `conditional_pass` → 附带条件继续

部分 Agent 支持 Review Panel 模式——多个子 Agent 并行审查，综合意见后由 Judge 裁决。整个过程无需人工干预，直到关键检查点才暂停等待审批。

#### 10.4 可视化组件体系

ACE 平台提供 6 种实时可视化视图，让工作流执行过程完全透明：

1. **总览面板**：实时状态卡片、转移次数统计、问题追踪
2. **时序图**：按时间线展示每次状态转移
3. **流转图**：状态访问次数、回退路径、热点分析
4. **Supervisor 视图**：每轮路由决策的时间线回放（方法、来源、目标、问题内容）
5. **Agent 流程图**：Agent 间消息传递拓扑
6. **状态图**：ReactFlow 拓扑图，实时高亮当前执行路径

### 11. 实战案例深挖

#### 11.1 Issue #701：AI 修复方案与人类专家完全一致

**问题描述：** 仓颉编译器 ICE（Internal Compiler Error）——泛型属性在 `extend` 块中被前向引用时，main 函数位置变化导致编译器崩溃（SIGILL，返回 -4），触发空指针解引用。

**工作流配置：** 状态机模式，5 个状态，18 个 Agent 角色，最多 50 次状态转移。

**执行过程：**

```
问题复现 → 根因定位 ⇄ 方案设计 → 代码修复 → 验证审计
              ↑          ↓
              └── 3次自动回滚 ──┘
```

1. **问题复现**：issue-reproducer 构造最小复现用例
2. **根因定位**：code-hunter 分析 + fix-breaker 挑战 + fix-judge 仲裁。沿 Sema/初始化检查等多条路径排查 ICE 与空指针类风险（CWE-476/CWE-617）
3. **方案设计**：architect 输出设计方案。根因与方案之间自动回退 3 次后才稳定推进
4. **人工审批**：方案设计后设置人工检查点，确认方向后再进入代码修复
5. **代码修复 + 验证**：按优化级别（O0/O1/O2）选择性回归测试

**关键亮点：长链路分析 vs 极简合入补丁**

分析阶段经历 57 步深度排查，但最终合入的生产代码极小——仅一行语义补丁：

```diff
 inline bool NotAssignableVariable(const VarDecl& vd, bool inInitFunction)
 {
     return !vd.isVar &&
         (vd.TestAnyAttr(Attribute::GLOBAL, Attribute::INITIALIZED, Attribute::ENUM_CONSTRUCTOR) ||
-            (vd.TestAnyAttr(Attribute::IN_STRUCT, Attribute::IN_CLASSLIKE) && !inInitFunction));
+            (vd.TestAnyAttr(Attribute::IN_STRUCT, Attribute::IN_CLASSLIKE, Attribute::IN_EXTEND) && !inInitFunction));
 }
```

**执行数据：**
- 总耗时 **16.5 小时**，完成 **57 个步骤**（含等待人工审查）
- 根因与方案设计之间 **自动回退 3 次**
- 生产代码变更：仅 `NotAssignableVariable` 单行语义补丁（`Attribute::IN_EXTEND`）
- 最终方案与人类专家方案 **完全一致**
- PR：https://gitcode.com/Cangjie/cangjie_compiler/pull/1293/diffs

**场景描述：** 为 OpenHarmony 开发仓颉 SDK API，涉及 C++ FFI 跨语言调用、API 设计文档、测试用例等多维度产出。

**Supervisor 路由实战示例：**

```
oh-cangjie-analyst 开始执行「Gap分析」
      │
      ├── Agent 评估信息缺口，输出：
      │   [NEED_INFO:human] 以下信息需要确认：
```

## 03-DFX Skills（场景化问题修复技能集）

### 1. 场景概述
- 场景名称：DFX Skills——仓颉诊断技能集
- 适用对象：仓颉编译器开发者、HarmonyOS 应用开发者、下游客户团队
- 发生频率：内存回归分析每次 SDK 升级均需执行
- 业务重要性：高——将专家经验固化为可复制的 skill，是能力外溢的关键载体

### 2. 背景与痛点
- 原来是怎么做的：内存回归分析依赖少数专家手动执行 hidumper、解析 smaps、对比 ELF section，流程长且易遗漏
- 原来的主要痛点：专家知识难以传递，新人无法独立完成；客户侧遇到同类问题只能求助原厂
- 为什么值得优化：将专家流程固化为 Agent 可执行的 skill，实现知识的规模化复制

### 3. AI 介入点
- AI 用在了哪个环节：Agent 按 skill 定义的流程自动执行诊断分析
- 具体承担什么角色：诊断执行者——按 SKILL.md 中定义的步骤和决策树自动执行
- 人工仍负责什么：数据采集（连接设备）、结论确认、优化方案决策

### 4. 操作流程（以内存回归分析为例）
- 第一步：数据采集——冷启动基线和目标设备，收集 hidumper 和 /proc/PID/smaps 数据
- 第二步：内存分类——`categorize_smaps.py` 将内存分为 20+ 类别（ArkTS Heap、Cangjie Heap、Jemalloc 等），输出差异表
- 第三步：增量定位——决策树判断：Jemalloc 增长 → 分析 native 分配；.so 膨胀 → `compare_so_sections.py` 对比 ELF section
- 第四步：根因验证——通过 GitCode API 关联 PR 变更，.so 替换测试隔离编译器 vs 运行时问题
- 第五步：量化报告——精确到字节级的增量归因和优化建议

**已覆盖的 skill 清单（6 个）：**

| 目录 | Skill 名称 | 功能 |
| --- | --- | --- |
| `cjc/memory-regression-analysis` | 内存回归分析 | 5 阶段流水线，从 smaps 分类到 ELF section 对比到根因定位 |
| `cjc/cangjie-oom-analyzer` | OOM 崩溃分析 | 解析 crash 日志 + cjheap dump，定位内存泄漏根因，生成支配树 SVG |
| `cjc/power-gitcode` | GitCode 平台操作 | 全功能 GitCode API 客户端（PR/Issue/Fork/Release 等） |
| `cjdb/` | cjdb 调试器 | 基于 pty 的 detached 调试会话，支持 Agent 跨轮次持久调试 |
| `cjpkg/` | 仓颉包管理 | 包搜索、下载、项目初始化、依赖配置 |
| `cjprof/` | CPU/堆性能分析 | CPU 采样热点报告、火焰图生成、堆内存泄漏分析 |

### 5. 使用工具与输入输出
- 使用工具：Claude Code + DFX Skills（纯 Python 脚本，仅依赖标准库）
- 输入材料：hidumper 输出、smaps 数据、.hap 文件（含 .so）、GitCode PR 信息、crash 日志、cjheap dump 文件、cjprof 采样数据
- 输出结果：结构化的诊断分析报告（内存回归报告、OOM 根因报告、CPU 热点火焰图、堆内存泄漏报告）
- 输出格式：终端输出 + Markdown 报告 + SVG 火焰图/支配树图

### 6. 实际效果
- 效率变化：从专家数天的手动分析缩短到 Agent 自动执行的数小时
- 质量变化：标准化流程避免遗漏，20+ 内存分类确保全面覆盖
- 使用者反馈：客户团队可独立使用，不再依赖原厂专家
- 是否形成复用价值：是——已推广到客户侧，验证了跨团队复制的可行性

### 7. 风险与限制
- 主要风险：设备连接和数据采集仍需人工操作，无法完全自动化
- 不适用情况：非 HarmonyOS 平台的内存分析
- 容易出错的地方：非冷启动场景下的内存数据可能不准确（skill 中已标注需使用 `--swap` 参数）

### 8. 经验总结
- 成功关键：
  - 每个 skill 自包含（SKILL.md + Python 脚本），零外部依赖，即插即用
  - 方法论编码为声明式流程，不仅提供工具还规定执行顺序和决策树
  - 分层文档设计（SKILL.md / PROMPT.md / REFERENCE.md）清晰分离关注点
- 复用建议：适合所有"专家流程可标准化"的诊断分析场景
- 后续优化方向：持续扩展诊断场景覆盖面

### 9. 一页摘要
- 这个案例解决了什么问题：专家知识难以传递和规模化复制的问题
- AI 的核心价值是什么：将专家经验固化为 Agent 可执行的 playbook，实现知识民主化
- 最值得推广的点是什么：skill 的自包含设计模式——SKILL.md + 脚本 + 分层文档
- 一句话结论：DFX Skills 让"只有专家能做"变成"任何人 + Agent 都能做"
