# 07-对外表达素材库

## 01-对外案例摘要

### 案例一：编译性能看板
- 对外名称：AI 驱动的编译性能可视化分析平台
- 问题概述：编译器团队缺乏前端技术栈，无法自建性能分析可视化工具
- 解决思路：团队成员担任产品经理，AI 担任全栈开发，快速搭建 Perfetto 风格的性能看板
- 实施方式：使用 Cursor AI 辅助，纯 HTML/CSS/JS + ECharts + Canvas 实现，零外部依赖
- 取得效果：一周内从零搭建约 5870 行代码的完整性能分析平台，支持时间线视图、内存曲线、阶段分析、离线导出
- 可复制启发："人当 PM、AI 当开发"的协作模式适用于所有"缺技术栈但有明确需求"的场景
- 脱敏说明：项目已开源，无需脱敏

### 案例二：ACE 平台
- 对外名称：ACE——以 Agent 为中心的研发工程平台
- 问题概述：个人 AI 辅助经验无法复制到团队，知识和 skill 分散在个人手中
- 解决思路：搭建平台化的 Agent 工作流引擎，让团队在统一平台上共建共用知识库和 skill
- 实施方式：Next.js 全栈应用，YAML 驱动的状态机工作流 + Supervisor-Lite 智能路由 + 攻防迭代机制
- 取得效果：Issue #701 自动修复 16.5 小时完成，方案与人类专家一致；已实现社区 issue 修复、PR 巡检、SDK 开发等工作流
- 可复制启发：状态机 + 攻防迭代 + 智能路由的架构模式可推广到其他研发团队
- 脱敏说明：项目已开源，无需脱敏

### 案例三：DFX Skills
- 对外名称：场景化诊断技能集——让专家经验人人可用
- 问题概述：内存回归分析等诊断工作高度依赖少数专家，知识传递困难
- 解决思路：将专家的诊断流程固化为 Agent 可执行的 skill，包含完整的步骤、决策树和自动化脚本
- 实施方式：纯 Python 标准库实现，每个 skill 自包含（SKILL.md + 脚本），即插即用
- 取得效果：客户团队可独立使用内存回归分析 skill，不再依赖原厂专家支持
- 可复制启发：skill 的自包含设计模式适用于所有"专家流程可标准化"的场景
- 脱敏说明：项目已开源，无需脱敏

## 03-标准说法与话术

### 1. 30 秒介绍
我们是仓颉编译器前端团队，从 2025 年 8 月开始系统性地探索 AI 辅助研发。我们不只是用 AI 写代码，而是搭建了以 Agent 为中心的开发平台（ACE），让团队的知识和经验可以持续沉淀、自动复用。目前已在社区 issue 修复、性能分析、内存诊断等场景落地，并将成熟的 skill 推广到客户侧。

### 2. 3 分钟介绍
我们团队面临需求过载、人才梯队不足、交付质量下滑的三重压力。2025 年 8 月，借仓颉开源和 Cursor 试用的契机，我们开始探索 AI 辅助研发。

摸索过程分四步：先用 Tab 补全提升个人编辑效率；再用 AI 消灭路径依赖，比如零前端经验搭建了编译性能看板；然后搭建 ACE 平台，把个人经验升级为团队共享的工作流和知识库；最后将成熟场景固化为 DFX Skills 推广到客户。

方法论上我们总结了五点：AI First-Try 思维、正确的人机关系（不要 PUA AI）、信息传递范式转换、识别协作边界、提供闭环场景。

核心成果：ACE 平台的 Issue #701 自动修复跑了 16.5 小时 57 步，最终方案与人类专家完全一致。这验证了 Agent 在编译器这类复杂系统工程中的可行性。

### 3. 常见问答
- 为什么从这些场景开始？——选择有明确输入输出、可自动验证的场景，降低试错成本
- 如何控制风险？——攻防迭代（AI 红队审查）+ 人工终审双重保障，所有合入代码必须通过 CI + 人工 review
- 如何判断是否值得推广？——看三个指标：成功率是否稳定、经验是否可固化为 skill、其他团队是否能独立使用
- AI 会不会产生错误代码？——会，这就是为什么我们设计了攻防迭代和人工审核机制，AI 是协作者不是替代者
