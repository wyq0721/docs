# 06-风险与治理

## 01-数据与信息安全

### 1. 关注的风险
- 敏感信息输入风险：编译器内部未开源的核心算法和优化策略可能被输入到外部模型
- 权限边界不清：团队成员对哪些代码可以输入 AI、哪些不可以缺乏明确指引
- 数据外泄担忧：使用外部 AI 服务时，代码片段可能被用于模型训练

### 2. 当前原则
- 可输入内容：已开源的仓颉编译器代码、公开的社区 issue、公开的技术文档
- 不可输入内容：未开源的核心算法、内部商业数据、客户敏感信息
- 脱敏要求：涉及内部系统的截图和日志需脱敏后使用

### 3. 控制措施
- 审核：ACE 平台的工作流产出需经人工 code review 后方可合入
- 脱敏：对外分享的案例和素材统一脱敏处理
- 使用规范：团队内建立 AI 使用规范文档，明确可用/不可用边界

## 02-输出可靠性与审核

### 1. 主要问题
- 幻觉：AI 可能生成看似合理但实际不正确的代码逻辑
- 事实错误：对仓颉语言特有语义的理解可能出现偏差
- 逻辑不严谨：复杂的编译器逻辑中，AI 可能遗漏边界条件
- 过度泛化：AI 倾向于给出通用方案，可能不适配编译器的特殊约束

### 2. 必须人工审核的内容
- 对外输出：所有提交到社区的 PR 必须经过人工 code review
- 涉及数据结论的内容：性能分析报告、内存回归分析结论
- 涉及业务判断的内容：架构设计方案、优化策略选择

### 3. 审核机制
- 谁审核：ACE 平台内置攻防迭代机制（Defender-Attacker-Judge），AI 产出先经 AI 红队审查，再经人工终审
- 审核什么：代码正确性、边界条件覆盖、与现有架构的一致性
- 审核标准：CI 门禁通过 + 人工 code review 通过

## 03-适用边界

### 1. 适合 AI 的工作
- 信息整理：代码库导航、模块结构梳理、API 文档生成
- 初稿生成：issue 修复方案初稿、测试用例生成、前端页面搭建
- 结构梳理：性能数据可视化、内存分类分析、ELF section 对比
- 模板化输出：GitCode PR/Issue 操作、commitlint 格式化、报告生成

### 2. 不适合完全交给 AI 的工作
- 最终决策：编译器架构方向、优化策略取舍
- 强专业判断：语言规范一致性审查、类型系统设计
- 高风险外部承诺：对客户的兼容性承诺、发布版本的质量兜底
- 敏感信息处理：涉及未开源代码的深度分析

### 3. 我们的边界判断
- 可辅助：所有研发环节均可让 AI 辅助，作为"第一稿"生成者
- 可主导初稿：社区 issue 修复、不兼容 PR 巡检、内存回归分析
- 必须人工主导：架构设计、发布决策、客户承诺

## 04-治理建议

### 1. 机制建议
- 建立统一使用规范：明确 AI 可用/不可用的代码和数据边界
- 建立案例复盘机制：每个 ACE 工作流运行后自动记录全链路日志，定期复盘
- 建立审核机制：攻防迭代 + 人工终审的双重保障

### 2. 能力建议
- 提升提示词能力：通过方法论沉淀（闭环场景、边界识别）提升团队的 AI 协作水平
- 提升人机协作能力：从"让 AI 干活"转变为"与 Agent 协作"，明确分工
- 提升复核能力：培养团队对 AI 产出的批判性审查意识

### 3. 推广建议
- 先试点后推广：每个新场景先在小范围验证，确认有效后再推广
- 先模板化后规模化：将成功经验固化为 skill，通过 ACE 平台分发
- 先控风险后提效率：确保审核机制到位后再扩大 AI 的自主权限
