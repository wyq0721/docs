# 02-推进时间线

## 01-启动阶段（2025 年 8-11 月）

### 1. 启动背景
- 为什么启动：Cursor 爆火引发关注，公司提供试用机会；仓颉语言同期开源，C++ 语料丰富，模型可直接理解
- 谁提出：团队核心成员在日常脚本编写中发现 AI 的效率优势，自发推动
- 当时的判断：AI 辅助开发在编译器领域有可行性，值得投入探索

### 2. 初始目标
- 想先验证什么：AI 能否在日常开发中切实提升效率，而非仅停留在 demo 层面
- 为什么先选这些场景：从最低门槛的 Tab 补全开始，风险最小，收益可感知

### 3. 启动方式
- 参与角色：个人先行探索
- 推进方式：在日常编码中使用 Cursor Tab 和 Copilot Tab
- 使用资源：公司提供的 Cursor 试用套餐

### 4. 初始产出
- 第一次输出了什么：代码编辑效率的直观提升，日常脚本编写基本可由 AI 代劳
- 得到了哪些反馈：团队成员开始关注并尝试

### 5. 启动阶段结论
- 哪些判断被验证：AI Tab 补全在编译器 C++ 代码上效果良好，语料充足是关键前提
- 哪些判断被修正：仅靠 Tab 补全的效率提升有天花板，需要更深层次的 AI 介入

---

## 02-探索阶段（2025 年 12 月 - 2026 年 1月）

### 1. 主要尝试
- 尝试 1：Tab 加速——在代码编辑期间通过 Cursor Tab 和 Copilot Tab 提升编辑效率
- 尝试 2：消灭路径依赖——把原本干不了或依赖周边协同的工作自己干起来，如编译性能看板（团队无前端技术栈，由 AI 实现）
- 尝试 3：探索 AI 在 issue 修复、代码分析等更复杂场景中的应用

### 2. 观察结果
- 哪些方向效果明显：消灭路径依赖效果显著——编译性能看板从零到可用仅一周，打破了"没有前端就做不了"的限制
- 哪些方向效果一般：纯 Tab 补全的效率提升有上限，约 20-30% 的编辑加速
- 哪些方向不适合继续：单兵作战、单点突破的方式无法对团队和组织带来进一步变革

### 3. 遇到的问题
- 工具层问题：模型对仓颉语言本身（非 C++ 部分）的理解有限
- 流程层问题：个人经验难以复制，缺乏统一的知识库和 skill 沉淀机制
- 组织协同问题：每个人的 AI 使用水平参差不齐，缺乏共享平台

### 4. 阶段性结论
- 有效经验：AI First-Try 思维有效，先让 AI 试，不行再人工介入
- 无效经验：期望 AI 一次性完美输出不现实，需要迭代协作
- 下一步调整方向：从单兵作战转向平台化，搭建团队共用的 Agent 平台

## 03-规模化阶段（2026 年 2 月 - 至今）

### 1. 为什么进入规模化
- 进入规模化的标志：ACE 平台搭建完成，团队开始在平台上共建共用
- 为什么认为值得推广：ACE 平台成功修复生产环境问题（如 Issue #701），验证了 Agent 工作流的可行性

### 2. 规模化动作
- 建立了哪些模板：ACE 工作流 YAML 模板（phase-based 和 state-machine 两种模式）、Agent 配置模板、Skill 目录结构模板
- 优化了哪些流程：从"人写代码 → 人 review"转变为"Agent 写代码 → AI 红队审查 → 人终审"的攻防迭代流程
- 推广了哪些做法：将内存回归分析、内存溢出等场景固化为 DFX Skills，推广到客户侧

### 3. 支撑机制
- 审核机制：ACE 平台内置 Defender-Attacker-Judge 攻防迭代，关键节点设置人工审批检查点
- 培训机制：通过 ACE 平台的知识库（覆盖 18 个模块、714 个文件、1036 个类）降低学习门槛，实现"代码即文档"
- 复盘机制：每次工作流运行全链路持久化（runs/{runId}/），支持事后分析和回放

### 4. 当前效果
- 哪些场景已具备复用价值：社区 issue 修复、不兼容 PR 巡检、编译器规格分析、小特性穿刺验证
- 哪些仍在试点：OpenHarmony SDK API 开发工作流、小特性自动分析与实现

## 04-当前状态与下一步

### 1. 当前状态
- 已完成：编译性能看板上线、ACE 平台核心功能开发、DFX Skills 首批 skill 发布
- 正在推进：ACE 平台的 Supervisor-Lite 智能路由优化、更多业务场景接入
- 尚未完成：组织级规模复制、完整的研发流程重塑

### 2. 当前成熟度判断
- 方法成熟度：中高——已形成 AI First-Try 思维和闭环场景方法论
- 组织成熟度：中——核心成员深度使用，全员普及仍在推进
- 工具成熟度：中高——ACE 平台功能完备，但自迭代能力仍需增强

### 3. 当前主要问题
- 问题 1：从单兵效率提升到组织效率提升的鸿沟尚未完全跨越
- 问题 2：业务断点识别不够系统化，如 PR 提交后如何自动知会检视人
- 问题 3：skill 的质量和覆盖度需要持续迭代

### 4. 下一步重点
- 短期：识别更多业务断点，补全工作流中的自动化缺口（如 PR 提交后自动知会检视人并附带根因及修复逻辑）
- 中期：规模复制，重塑流程——基于 ACE 平台实现从个人效率提升到组织研发效率提升
- 长期（预判与建议）：
  - ACE 平台自迭代闭环：每次工作流运行后自动更新知识库和 skill，让平台越用越聪明，减少人工维护成本
  - 度量驱动的持续优化：建立 AI 辅助研发的效能度量体系（成功率、修复周期、人工介入率等），用数据指导场景优先级和资源投入
  - 跨团队生态建设：从仓颉编译器前端团队扩展到编译器全链路乃至更广泛的 HarmonyOS 开发者生态，ACE 平台和 DFX Skills 成为公共基础设施
  - Agent 间协作深化：随着 MCP 等协议成熟，探索跨平台、跨组织的 Agent 协作，如 ACE 平台与 CI/CD 系统、代码审查系统的深度集成
