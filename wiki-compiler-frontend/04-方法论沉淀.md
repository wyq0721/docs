# 04-方法论沉淀

## 01-场景选择方法

### 1. 什么样的场景值得做
- 重复性高：社区 issue 修复、不兼容 PR 巡检等周期性任务
- 规则相对清晰：有明确的输入（issue 描述、PR diff、smaps 数据）和判断标准
- 有标准输入输出：如 issue → 修复 PR、性能数据 → 可视化看板、smaps → 分类对比报告
- 对效率提升敏感：这些任务占用大量人力，提效后可释放团队产能

### 2. 什么样的场景暂时不建议做
- 高度依赖专家判断：编译器架构设计、优化策略选择等需要深度领域经验的决策
- 对事实准确性要求极高且难复核：涉及语言规范一致性的判断
- 涉及强敏感数据：内部未开源的核心算法和商业数据
- 缺乏稳定输入材料：需求尚未明确、规格频繁变动的探索性工作

### 3. 场景筛选标准
- 价值大小：能释放多少人力、对交付质量的影响程度
- 落地难度：是否有明确的输入输出、是否可自动化验证
- 可复用性：是否可固化为 skill 推广到其他团队或客户
- 风险等级：出错后的影响范围和修复成本

### 4. 我们的筛选方法
- 谁提出场景：团队任何成员均可提出，鼓励 AI First-Try
- 谁评估：团队核心成员评估可行性和优先级
- 谁决定试点：场景提出者即为试点负责人

| 场景 | 价值 | 难度 | 风险 | 是否试点 | 原因 |
| --- | --- | --- | --- | --- | --- |
| 编译性能看板 | 高 | 中 | 低 | 是 | 团队无前端能力但急需可视化分析工具 |
| 社区 issue 自动修复 | 高 | 高 | 中 | 是 | 占用大量人力且有明确验证标准（CI 通过） |
| 不兼容 PR 巡检 | 高 | 中 | 低 | 是 | 规则清晰，适合自动化 |
| 内存回归分析 | 高 | 中 | 低 | 是 | 流程固定，可固化为 skill |
| 小特性分析与实现 | 中 | 高 | 中 | 是 | 验证 Agent 在复杂任务上的能力边界 |

## 02-推进落地方法

### 1. 推进思路
- 总体原则：AI First-Try——任何任务先让 AI 试，不行再人工介入
- 为什么采用这种推进方式：降低心理门槛，鼓励团队大胆尝试，在实践中积累经验

### 2. 典型路径
- 小范围试点：个人先行尝试 Tab 加速和简单脚本生成
- 快速验证：选择一个具体场景（如编译性能看板）端到端验证 AI 可行性
- 复盘优化：总结经验教训，提炼方法论（如闭环场景、边界识别）
- 模板沉淀：将成功经验固化为 ACE 工作流模板和 DFX Skill
- 扩大使用：通过 ACE 平台分发给团队，通过 DFX Skills 推广到客户

### 3. 角色分工
- 发起人：任何团队成员（AI First-Try，人人可发起）
- 使用者：团队全员
- 审核者：领域专家 + ACE 平台的 Judge Agent
- 推动者：团队核心成员，负责平台建设和方法论沉淀

### 4. 成功条件
- 有真实业务场景：不做 demo，只做真实业务中的痛点
- 有明确负责人：场景提出者即为负责人
- 有复盘机制：ACE 平台全链路日志 + 定期复盘
- 有最小闭环：每个场景必须有可验证的输入输出和验收标准

## 03-提示词与人机协作

### 1. 提示词使用原则（五大方法论）

#### 方法论一：AI First-Try 思维（道）
- 核心理念：任何任务先让 AI 试，不行再人工介入
- 这不是偷懒，而是一种思维转变——把 AI 当作团队的"第一响应者"
- 好处：降低尝试门槛，快速发现 AI 的能力边界，积累协作经验
- 实践：团队内建立共识，遇到任务第一反应是"AI 能不能帮忙"，而非"这个太复杂了 AI 肯定不行"

#### 方法论二：谁调教谁？
- 常见误区：试图"PUA"AI，用强硬指令逼它产出完美结果
- 现实：你 PUA 它，它给你制造幻觉——越是强迫它"必须正确"，它越倾向于编造看似正确的答案
- 正确姿势：与 AI 建立协作关系而非主从关系，给它足够的上下文和退路（允许它说"不确定"）
- 关键认知：AI 不是下属，不会因为你态度强硬就产出更好的结果

#### 方法论三：信息传递的范式转换
- 传统模式：人与人之间传递需求，依赖口头沟通、文档、会议
- 新模式：人与 Agent、Agent 与 Agent 之间传递信息
- 核心挑战：澄清需求的重要性被放大了——人与人之间可以靠默契和追问补齐信息，但 Agent 不会主动追问（除非你设计了这个机制）
- ACE 的解法：Supervisor-Lite 的 `[NEED_INFO]` 协议让 Agent 可以主动声明信息缺口，由路由器决定问谁

#### 方法论四：识别边界
- 不要把 AI 当全能神，也不要把它当小白
- 找到与 Agent 协作的边界，就跟团队内人与人合作一样——你不会把所有工作都甩给一个同事，也不会事无巨细地指导每一步
- 实践经验：
  - AI 擅长：信息检索、代码生成初稿、模式匹配、重复性任务
  - AI 不擅长：架构决策、创造性设计、跨模块的全局影响评估
  - 最佳协作点：人做方案和审计，AI 做执行和初稿

#### 方法论五：提供闭环场景
- 为 Agent 分配任务时，一定要明确规格和可测试的验收目标
- 跟领导给你派活是一样的——如果领导只说"把这个搞好"，你也不知道怎么干
- 闭环场景的要素：
  - 明确的输入：issue 描述、代码上下文、相关文档
  - 明确的输出：修复 PR、分析报告、可视化页面
  - 可验证的标准：CI 通过、测试用例通过、人工 review 通过
- ACE 的实践：每个工作流步骤都有明确的 task 描述和 Judge 评判标准

### 2. 常见提示词结构
- 背景：当前代码库的模块结构、相关文件路径
- 目标：需要修复的 issue 或实现的功能
- 输入材料：issue 描述、错误日志、相关代码片段
- 输出要求：修复代码 + 测试用例 + 修改说明
- 注意事项：不要修改无关文件、遵循现有代码风格

提示词模板：

```text
你现在要扮演：仓颉编译器前端开发工程师
任务目标是：修复 Issue #XXX（简述问题）
背景信息如下：（模块结构、相关文件、已知约束）
请基于以下材料处理：（issue 描述、错误复现步骤、相关代码）
输出格式要求：（修复代码 + 测试用例 + 根因分析）
注意不要：（修改无关文件、引入新依赖、破坏现有 API）
```

### 3. 人机协作分工
- AI 更适合：代码生成初稿、issue 根因定位、测试用例生成、文档生成、跨语言/跨领域的快速原型
- 人更适合：方案设计、架构决策、代码审计、最终质量把关
- 必须人工复核的环节：所有合入主干的代码、对外发布的文档、涉及兼容性的变更

### 4. 常见问题
- 输出空泛：输入上下文不足，AI 只能给出通用建议——解法：提供具体的代码片段和约束条件
- 事实不准：AI 对仓颉语言特有语义理解有限——解法：在 prompt 中提供相关规范说明
- 结构看似完整但不可用：AI 生成的代码编译不过或逻辑有误——解法：提供闭环场景，让 CI 自动验证

## 04-评估与复盘机制

### 1. 我们评估什么
- 效率：任务完成时间的变化（如 issue 修复从人工数天到 Agent 数小时）
- 质量：修复的正确性（CI 通过率、人工 review 通过率）
- 稳定性：同类任务的成功率是否稳定
- 可复用性：经验是否已固化为 skill 或工作流模板

### 2. 如何评估
- 使用前后对比：如 Issue #701 人工预估需数天，Agent 16.5 小时完成且方案一致
- 使用者反馈：团队成员对 AI 辅助的主观体验和建议
- 样本复盘：定期抽取 ACE 工作流运行记录进行深度分析
- 输出质量抽检：对 Agent 产出的代码进行专项 review

### 3. 复盘方式
- 复盘频率：每个重要工作流运行后即时复盘，团队层面月度复盘
- 参与角色：场景负责人 + 领域专家
- 复盘产出：经验总结、方法论更新、skill 优化建议

### 4. 复盘后如何沉淀
- 是否形成模板：成功的工作流配置固化为 YAML 模板
- 是否更新方法：将新发现的最佳实践更新到方法论文档
- 是否决定推广或停止：基于成功率和 ROI 决定是否推广到更多场景

## 05-常见误区与避坑

### 1. 常见误区
- 把 AI 当成完全替代人工：AI 是协作者不是替代者，人工审核不可省略
- 只看演示效果，不看真实流程：demo 中的效果和生产环境差距很大，必须在真实场景中验证
- 输入质量不足却期待高质量输出：garbage in, garbage out，上下文不足时 AI 只能猜测
- 没有审核机制就直接使用结果：编译器代码的正确性要求极高，必须有 CI + 人工双重把关

### 2. 我们踩过的坑
- 坑 1：早期试图用强硬 prompt "逼"AI 产出完美代码，结果 AI 制造更多幻觉——改为给足上下文、允许不确定
- 坑 2：单兵作战积累的经验无法复制给团队——改为搭建 ACE 平台，将经验固化为工作流和 skill
- 坑 3：给 Agent 的任务描述过于模糊，产出质量不稳定——改为提供闭环场景，明确输入输出和验收标准

### 3. 对应改进措施
- 改进 1：建立"与 AI 协作"而非"指挥 AI"的心智模型
- 改进 2：通过 ACE 平台实现经验的平台化沉淀和分发
- 改进 3：每个工作流步骤必须定义明确的 task、输入、输出和 Judge 标准

### 4. 给后来者的建议
- 建议 1：从 AI First-Try 开始，降低心理门槛，先用起来再优化
- 建议 2：尽早搭建团队共享平台，避免个人经验孤岛
- 建议 3：重视闭环场景设计，这是 AI 产出质量的关键杠杆
